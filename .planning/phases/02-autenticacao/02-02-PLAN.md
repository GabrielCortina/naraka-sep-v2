**1. Create `src/features/auth/components/login-form.tsx`** (per D-04, D-05, D-06):

Client component (`'use client'`) implementing the login form per UI-SPEC specifications:

**Structure:**
- Form with two fields: nome (text) and pin (password with numeric input mode)
- Submit button "Entrar" with loading state
- Inline error message below form

**Form fields:**
- Nome field: `<Label htmlFor="nome">Nome</Label>` + `<Input id="nome" type="text" placeholder="Seu nome" autoComplete="username" required />` with min 2 chars
- PIN field: `<Label htmlFor="pin">PIN</Label>` + `<Input id="pin" type="password" inputMode="numeric" pattern="[0-9]*" maxLength={6} minLength={4} placeholder="Digite seu PIN" autoComplete="current-password" required />`

**State management:**
- `nome` (string), `pin` (string), `error` (string | null), `loading` (boolean)
- Error clears when user modifies either field (`onChange` handlers set `setError(null)`)

**Submit handler (per D-03):**
1. Set loading=true, error=null
2. Import `nomeToEmail` from `@/features/auth/lib/slugify`
3. Convert nome to email: `const email = nomeToEmail(nome)`
4. Call `supabase.auth.signInWithPassword({ email, password: pin })`
5. On error: set error to "Nome ou PIN incorreto" (D-06), set loading=false, focus nome field
6. On success: read `returnTo` from URL search params. If present and starts with '/', redirect there. Otherwise get role from session JWT claims and redirect to ROLE_DEFAULTS[role]. Use `router.push()` from next/navigation.
   - To read role from session after login: decode the access_token JWT payload (base64 decode the middle segment) and read `user_role` field. No need for jose here since we just logged in (trust the fresh token client-side).

**Loading state (per UI-SPEC):**
- Button disabled, shows Lucide `Loader2` icon with `animate-spin` class + text "Entrando..."

**Accessibility (per UI-SPEC):**
- Labels associated with inputs via htmlFor/id
- Error message: `role="alert" aria-live="polite"`
- Enter key submits form from any field (native form behavior)
- After failed login, focus returns to nome field via `nomeInputRef.current?.focus()`

**Styling (per UI-SPEC):**
- All inputs minimum 44px height on mobile (h-11 in Tailwind)
- Button full width (w-full), minimum 44px height (h-11)
- Form fields spaced with gap-6 (24px per spacing scale lg)

**Rewrite `middleware.ts`** to extend the existing session refresh with JWT verification and role-based route protection (per D-10, D-11, AUTH-03, AUTH-04):

**Import additions:**
```typescript
import { jwtVerify } from 'jose'
```

**Note on imports:** The middleware runs in Edge Runtime. Do NOT import from `src/features/auth/lib/role-config.ts` because path aliases may not resolve in Edge middleware. Instead, inline the ROLE_ROUTES and ROLE_DEFAULTS constants directly in middleware.ts. Copy the exact values:
```typescript
const ROLE_ROUTES: Record<string, string[]> = {
  admin: ['/dashboard', '/upload', '/fardos', '/prateleira', '/baixa'],
  lider: ['/dashboard', '/upload', '/fardos', '/prateleira'],
  separador: ['/prateleira'],
  fardista: ['/fardos', '/baixa'],
}

const ROLE_DEFAULTS: Record<string, string> = {
  admin: '/dashboard',
  lider: '/dashboard',
  separador: '/prateleira',
  fardista: '/fardos',
}
```

**Middleware logic flow:**

1. **Keep existing Supabase session refresh** (createServerClient with cookie getAll/setAll, call `supabase.auth.getUser()`).

2. **Determine the current route path:** `const pathname = request.nextUrl.pathname`

3. **Skip auth for public routes:** If pathname is `/login`, allow through (return supabaseResponse). Also skip if pathname starts with `/api/` (API routes handle their own auth).

4. **Get session:** `const { data: { session } } = await supabase.auth.getSession()`

5. **No session -> redirect to login (D-11):**
   ```typescript
   if (!session) {
     const url = request.nextUrl.clone()
     url.pathname = '/login'
     // Only set returnTo if not already on login and path is meaningful
     if (pathname !== '/' && pathname !== '/login') {
       url.searchParams.set('returnTo', pathname)
     }
     return NextResponse.redirect(url)
   }
   ```

6. **Verify JWT and extract role (D-10):**
   ```typescript
   try {
     const secret = new TextEncoder().encode(process.env.SUPABASE_JWT_SECRET!)
     const { payload } = await jwtVerify(session.access_token, secret)
     const userRole = payload.user_role as string | undefined
   ```
   If `jwtVerify` throws or `userRole` is null/undefined, redirect to `/login` (invalid session).

7. **Root path redirect:** If pathname is `/`, redirect to `ROLE_DEFAULTS[userRole]`.

8. **Check route permission (D-10):**
   ```typescript
   const allowedRoutes = ROLE_ROUTES[userRole]
   if (allowedRoutes && !allowedRoutes.some(route => pathname.startsWith(route))) {
     return NextResponse.redirect(new URL(ROLE_DEFAULTS[userRole] || '/login', request.url))
   }
   ```

9. **Return supabaseResponse** (pass through with refreshed cookies).

**Matcher config** (keep existing but ensure login is included for proper session handling):
```typescript
export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon.ico).*)'],
}
```

**Edge Runtime note:** Do NOT add `export const runtime = 'edge'` -- Next.js middleware is Edge by default. Only use `jose` (not `jsonwebtoken`), only use `TextEncoder` (not `Buffer`).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> middleware | Untrusted cookies cross into JWT verification |
| login form -> Supabase Auth | User-supplied nome+PIN sent to authentication |
| middleware -> JWT claims | Token claims used for authorization decisions |
| returnTo param -> redirect | User-supplied URL used for post-login redirect |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | middleware JWT | mitigate | Verify JWT signature with jose + SUPABASE_JWT_SECRET before trusting claims |
| T-02-05 | Spoofing | login form | mitigate | Generic error message "Nome ou PIN incorreto" prevents user enumeration (D-06) |
| T-02-06 | Spoofing | returnTo param | mitigate | Validate returnTo starts with '/' and does not contain protocol (prevent open redirect) |
| T-02-07 | Elevation of Privilege | route access | mitigate | Middleware checks ROLE_ROUTES server-side on every request; client-side filtering is UX only |
| T-02-08 | Denial of Service | login form | accept | Supabase Auth has built-in rate limiting; no custom rate limiting needed |
</threat_model>
