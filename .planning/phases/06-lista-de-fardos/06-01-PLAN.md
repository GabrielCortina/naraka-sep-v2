    **Logica de fetch (fetchFardos callback):**
    1. Query `reservas` com status='reservado' — esses sao os fardos pendentes da lista
    2. Query `trafego_fardos` — fardos ja encontrados (status='encontrado') ou nao encontrados
    3. Query `atribuicoes` com tipo='fardista' JOIN users(nome) — quem esta atribuido a cada card
    4. Query `atribuicoes` com tipo='separador' JOIN users(nome) — separador de cada card (para PDF D-27)
    5. Query `fardos_nao_encontrados` — para marcar quais fardos foram N/E

    **Montagem da lista plana (D-01):**
    - Para cada reserva com status='reservado': criar FardoItem
    - Derivar card_key: buscar em `pedidos` quais cards usam este SKU + importacao_numero. Usar o primeiro card_key encontrado (reserva vinculada ao SKU, D-09 da Phase 4)
    - Para isso, tambem fazer query de `pedidos` com select('sku, card_key, importacao_numero')
    - Derivar status: se trafego_fardos tem registro com este codigo_in e status='encontrado' => 'encontrado'. Se fardos_nao_encontrados tem registro => 'nao_encontrado'. Senao => 'pendente'
    - Derivar fardista_nome: via atribuicoes (tipo='fardista') para o card_key do fardo
    - Derivar separador_nome: via atribuicoes (tipo='separador') para o card_key do fardo
    - is_cascata: false para todos nesta phase (cascata vem da Phase 7)

    **Filtro por role (D-04, D-05):**
    - Se userRole === 'fardista': filtrar apenas fardos cujo card_key tem atribuicao para este userId com tipo='fardista'
    - Se userRole === 'admin' ou 'lider': mostrar todos

    **Counters (D-09):**
    - Calcular FardoCounters a partir da lista completa (antes de filtros de busca/status/atribuicao)

    **Realtime (FARD-04):**
    - Chamar `useCardsRealtime(fetchFardos)` para re-fetch automatico em mudancas

    **Retorno:**
    ```typescript
    return { fardos, counters, loading, error, refetch: fetchFardos }
    ```

    Usar `Array.from()` para qualquer iteracao de Map (padrao tsconfig do projeto).
