# Phase 07.1: Aba de Transformacao - Research

**Researched:** 2026-04-08
**Domain:** Next.js 14 feature page (new route + Supabase realtime + reusable kanban components)
**Confidence:** HIGH

## Summary

This phase adds a dedicated `/transformacao` tab for managing items sent to transformation (when cascata finds no available bale). The implementation is heavily pattern-based -- the existing Prateleira page (`prateleira-client.tsx`) serves as an almost 1:1 blueprint. The core work involves: (1) a DB migration adding `numero_transformacao` column, (2) a new feature folder `src/features/transformacao/` with adapted components, (3) new API routes for fetching/confirming/assigning transformacao items, (4) navigation changes in role-config + middleware + bottom-tabs, and (5) Supabase realtime subscriptions for live updates.

The key technical challenge is the sequential card logic (D-11): when a completed transformacao card receives new items, a NEW card with incremented `numero_transformacao` must be created rather than reopening the old one. This requires grouping transformacao records by `card_key + numero_transformacao` and computing the next sequence number on INSERT.

**Primary recommendation:** Follow the Prateleira pattern exactly -- new feature folder, custom hook for data fetching, adapted client component, reuse KanbanBoard/AssignModal/CompletedSection/NumpadPopup as-is, create TransformacaoCard/TransformacaoModal/TransformacaoHeader as new components.

### Pattern 1: Data Fetching & Assembly (from use-card-data.ts)
**What:** Custom hook fetches transformacoes table, groups by `card_key + numero_transformacao`, computes card-level aggregates (progress, status, assignment).
**When to use:** The transformacao data hook.
**Key difference from use-card-data:** Reads from `transformacoes` table directly instead of assembling from pedidos+progresso+reservas. Each transformacao row IS an item.

### Pattern 4: Sequential Card Number Logic (D-11, CRITICAL)
**What:** When cascata INSERTs a new transformacao record, it must compute `numero_transformacao` based on whether the current transformacao card for that card_key is completed or active.
**Implementation approach:**

1. **On INSERT (cascata route):** Query `MAX(numero_transformacao)` for the card_key. Check if ALL items with that max number are `concluido`. If yes, use `max + 1`. If no (active card exists), use `max` (same card).
2. **On frontend:** Group transformacoes by `card_key + numero_transformacao` to create separate cards.

```sql
-- Migration: add numero_transformacao
ALTER TABLE transformacoes
  ADD COLUMN numero_transformacao INTEGER NOT NULL DEFAULT 1;
```

```typescript
// Source: [ASSUMED - logic derived from D-11]
// In cascata route, before INSERT:
const { data: maxRow } = await supabaseAdmin
  .from('transformacoes')
  .select('numero_transformacao, status')
  .eq('card_key', card_key)
  .order('numero_transformacao', { ascending: false })
  .limit(1)

let nextNum = 1
if (maxRow && maxRow.length > 0) {
  const currentMax = maxRow[0].numero_transformacao
  // Check if all items in current max are concluido
  const { data: currentItems } = await supabaseAdmin
    .from('transformacoes')
    .select('status')
    .eq('card_key', card_key)
    .eq('numero_transformacao', currentMax)
  
  const allDone = currentItems?.every(i => i.status === 'concluido')
  nextNum = allDone ? currentMax + 1 : currentMax
}
```

## Common Pitfalls

### Pitfall 1: Middleware ROLE_ROUTES Desync
**What goes wrong:** Adding /transformacao to role-config.ts but forgetting the inlined copy in middleware.ts. Separadores get redirected away from /transformacao.
**Why it happens:** Middleware uses Edge Runtime which cannot resolve path aliases, so ROLE_ROUTES is copy-pasted.
**How to avoid:** Update BOTH files in the same task. Verify with `middleware.ts` line 5-9.
**Warning signs:** Role-based access works in dev (client-side) but not on refresh (server middleware redirects).

### Pitfall 2: NumpadPopup Validation Change
**What goes wrong:** The existing NumpadPopup allows `qty >= 0 && qty <= quantidadeNecessaria`. Transformacao requires EXACT match (`qty === quantidadeNecessaria`).
**Why it happens:** Transformacao is "tudo ou nada" (D-06), unlike Prateleira which allows partial.
**How to avoid:** Do NOT modify NumpadPopup globally. Instead, validate in the TransformacaoModal's confirm handler AFTER numpad returns. If quantity !== required, show toast error and don't process.
**Warning signs:** Separador can confirm partial quantities in transformacao.

### Pitfall 3: Realtime Channel Name Collision
**What goes wrong:** Using the same channel name as existing subscriptions causes events to be mixed or dropped.
**Why it happens:** Supabase channels must have unique names per client.
**How to avoid:** Use distinct channel name like `'transformacao-realtime'` (not `'cards-realtime'`).
**Warning signs:** Transformacao page causes prateleira realtime to break or vice versa.

### Pitfall 4: Sequential Number Race Condition
**What goes wrong:** Two concurrent cascata requests for the same card_key both read the same MAX(numero_transformacao) and both assign the same number.
**Why it happens:** No DB-level constraint enforcing uniqueness of (card_key, numero_transformacao) per status group.
**How to avoid:** The cascata route already runs server-side with supabaseAdmin (single execution context). For extra safety, compute next number in SQL or use a transaction. In practice, cascata is triggered by user action (one at a time per card), so collision is unlikely.
**Warning signs:** Two transformacao cards with the same numero_transformacao for the same card_key.

### Pitfall 5: KanbanBoard Renders OrderCard Hardcoded
**What goes wrong:** KanbanBoard -> KanbanColumn -> OrderCard is hardcoded. Cannot render TransformacaoCard without changes.
**Why it happens:** KanbanColumn imports and renders OrderCard directly.
**How to avoid:** Two approaches: (A) Create a TransformacaoKanbanBoard/Column that renders TransformacaoCard, or (B) Make KanbanColumn accept a card renderer prop. Approach A is simpler and avoids touching shared components.
**Warning signs:** Transformacao cards look identical to prateleira cards (missing purple badge, missing sequential number).

### Pitfall 6: Forgetting grupo_envio Derivation
**What goes wrong:** The `transformacoes` table has `card_key` but NOT `grupo_envio` or `tipo` or `importacao_numero`. Kanban columns need grupo_envio to group.
**Why it happens:** card_key format is `{grupo_envio}|{tipo}|{importacao_numero}` but this must be parsed or joined with pedidos.
**How to avoid:** Parse card_key to extract grupo_envio, tipo, importacao_numero. OR join with pedidos table. Parsing is simpler since the format is consistent.
**Warning signs:** All transformacao cards appear in one column or columns are blank.

1. **card_key format parsing** (RESOLVED)
   - What we know: card_key is used as grouping key throughout the system, constructed in card-utils groupByCardKey
   - **Resolution:** Confirmed format is `{grupo_envio}|{tipo}|{importacao_numero}` -- verified in `src/features/upload/lib/classify.ts:36` (`generateCardKey` uses template literal with `|` delimiter). Parse with `cardKey.split('|')`.

2. **Existing transformacoes data** (RESOLVED)
   - What we know: The cascata route already inserts into transformacoes without numero_transformacao
   - **Resolution:** Migration uses `DEFAULT 1` which correctly assigns numero_transformacao=1 to any existing rows. This is the correct behavior since all pre-migration rows belong to "Transformacao #1" for their respective card_key.
