# Phase 07.1: Aba de Transformacao - Research

**Researched:** 2026-04-08 (re-research)
**Domain:** Next.js 14 feature page (new route + Supabase realtime + reusable kanban components)
**Confidence:** HIGH

## Summary

This phase adds a dedicated `/transformacao` tab for managing items sent to transformation (when cascata finds no available bale). The implementation is heavily pattern-based -- the existing Prateleira page (`prateleira-client.tsx`) serves as an almost 1:1 blueprint. The core work involves: (1) a DB migration adding `numero_transformacao` column, (2) updating the cascata INSERT to populate `numero_transformacao`, (3) a new feature folder `src/features/transformacao/` with adapted components, (4) new API routes for fetching/confirming/assigning transformacao items, (5) navigation + middleware changes to add `/transformacao`, and (6) Supabase realtime subscriptions for live updates.

The key technical challenge is the sequential card logic (D-11): when a completed transformacao card receives new items, a NEW card with incremented `numero_transformacao` must be created. This requires the cascata route to compute `MAX(numero_transformacao) + 1` for a given `card_key` when all existing transformacoes for that card_key are `concluido`. The frontend groups transformacao records by `card_key + numero_transformacao` to display separate cards.

**Primary recommendation:** Follow the Prateleira pattern exactly -- new feature folder, custom hook for data fetching, adapted client component, reuse KanbanBoard/AssignModal/CompletedSection/NumpadPopup as-is, create TransformacaoCard/TransformacaoModal/TransformacaoHeader as new components.

### Pattern 1: Feature Data Hook (use-transformacao-data.ts)

**What:** Custom hook that fetches transformacoes + atribuicoes, groups by card_key + numero_transformacao, applies role-based visibility, and returns assembled card data.
**When to use:** For the main data loading of the /transformacao page.
**Source:** Adapted from `src/features/cards/hooks/use-card-data.ts` [VERIFIED: codebase]

Key differences from use-card-data:
1. Fetches from `transformacoes` table instead of `pedidos` + `progresso`
2. Groups by `card_key + numero_transformacao` (compound key) instead of just `card_key`
3. No reservas lookup (transformacao items have no fardo reservations)
4. Atribuicoes still fetched for assignment info (same pattern)
5. Urgency computed from original card's grupo_envio (available on pedidos via card_key)

### Pattern 2: Strict Quantity Validation (TransformacaoModal)

**What:** NumpadPopup opens with quantidadeNecessaria = item.quantidade. On confirm, if entered value !== quantidadeNecessaria exactly, show error toast and keep numpad open.
**When to use:** For the single confirmation action in TransformacaoModal.
**Source:** NumpadPopup already validates `qty <= quantidadeNecessaria`. For transformacao, change to `qty === quantidadeNecessaria`. [VERIFIED: numpad-popup.tsx line 49]

Current NumpadPopup line 49: `if (qty >= 0 && qty <= quantidadeNecessaria)` -- this allows partial. For transformacao, the parent component should validate strict equality AFTER numpad confirms, showing error toast if mismatch. This avoids modifying the shared NumpadPopup.

**Recommended approach:** Let NumpadPopup work as-is (accepts any qty <= max). In TransformacaoModal's handleConfirm callback, check `qty !== item.quantidade` and if mismatch, show `toast.error("Quantidade deve ser exatamente {N}")` and do not call API. This is cleaner than forking NumpadPopup.

### Pattern 3: Sequential Card Number (numero_transformacao)

**What:** Each transformacao record has a `numero_transformacao` INTEGER. When cascata inserts a new record, it computes the correct number:
- If no existing transformacoes for this card_key, numero = 1
- If all existing transformacoes for this card_key are `concluido`, numero = MAX(numero_transformacao) + 1
- If there are `pendente`/`atribuido` items for this card_key, numero = MAX(numero_transformacao) (add to existing open card)

**Where to implement:** In `app/api/prateleira/cascata/route.ts` lines 286-297, before the INSERT statement.
**Source:** D-11, D-12, D-14 from CONTEXT.md [VERIFIED: CONTEXT.md]

```typescript
// Compute numero_transformacao for the new record
const { data: existingTransf } = await supabaseAdmin
  .from('transformacoes')
  .select('numero_transformacao, status')
  .eq('card_key', card_key)
  .order('numero_transformacao', { ascending: false })

let numero = 1
if (existingTransf && existingTransf.length > 0) {
  const maxNumero = existingTransf[0].numero_transformacao
  const allCompleted = existingTransf.every(t => t.status === 'concluido')
  numero = allCompleted ? maxNumero + 1 : maxNumero
}
```

### Pattern 4: Realtime Subscription for Transformacoes

**What:** Supabase channel subscribing to postgres_changes on `transformacoes` table for INSERT/UPDATE events.
**Source:** `use-cards-realtime.ts` already subscribes to transformacoes table [VERIFIED: codebase line 37-39]. For the transformacao page, create a dedicated hook that triggers refetch.

```typescript
// use-transformacao-realtime.ts
export function useTransformacaoRealtime(onUpdate: () => void) {
  useEffect(() => {
    const supabase = createClient()
    const channel = supabase
      .channel('transformacao-realtime')
      .on('postgres_changes', { event: '*', schema: 'public', table: 'transformacoes' }, () => onUpdate())
      .on('postgres_changes', { event: '*', schema: 'public', table: 'atribuicoes' }, () => onUpdate())
      .subscribe()
    return () => { supabase.removeChannel(channel) }
  }, [onUpdate])
}
```

### Anti-Patterns to Avoid

- **Forking NumpadPopup:** Do NOT create a separate strict-numpad component. Use the existing one and validate in the parent callback.
- **Forking KanbanBoard:** Do NOT modify KanbanBoard. Create TransformacaoCard that wraps/adapts to CardData interface, then pass to KanbanBoard as-is.
- **Mutating completed cards:** D-14 is absolute -- never update a transformacao record with status='concluido'. New items always get a new numero_transformacao.
- **Subscribing in modal:** Do NOT create a realtime subscription inside TransformacaoModal. The parent (transformacao-client.tsx) should handle realtime, and pass updated data down via props/state, same as Prateleira pattern where selectedCard syncs via useEffect.

## Common Pitfalls

### Pitfall 1: Middleware ROLE_ROUTES out of sync
**What goes wrong:** Adding /transformacao to role-config.ts but forgetting middleware.ts inlined copy. Middleware blocks access for separador.
**Why it happens:** middleware.ts has its own ROLE_ROUTES because Edge Runtime cannot resolve path aliases. Two separate copies must stay in sync.
**How to avoid:** Update both files in the same task/commit. Verify both have identical route arrays.
**Warning signs:** User gets redirected to default page when navigating to /transformacao.

### Pitfall 2: KanbanBoard expects CardData with urgency computed from grupo_envio
**What goes wrong:** Transformacao cards lack proper urgency because they don't have pedidos-based deadline data.
**Why it happens:** Transformacao records have card_key but no direct prazo_envio. Urgency must be derived from the original card's grupo_envio.
**How to avoid:** The use-transformacao-data hook must join transformacoes.card_key with pedidos to get grupo_envio and compute urgency via getUrgencyTier.
**Warning signs:** All transformacao cards show same urgency tier, or no countdown.

### Pitfall 3: Realtime modal update race condition
**What goes wrong:** New item arrives via realtime while user is mid-confirmation. The refetch resets selectedCard state, closing the numpad or losing context.
**Why it happens:** The parent refetches all data on realtime event, which causes selectedCard to update.
**How to avoid:** Same pattern as Prateleira: `useEffect` that syncs selectedCard with updated cards list. The key is to only update the data, not close the modal. Prateleira-client.tsx lines 142-148 show this exact pattern.
**Warning signs:** Modal flickers or closes when another item arrives.

### Pitfall 4: numero_transformacao race condition on concurrent cascata calls
**What goes wrong:** Two cascata calls for the same card_key happen simultaneously. Both read MAX(numero_transformacao) = 1, both insert with numero = 2 (or both with 1).
**Why it happens:** The SELECT + INSERT in cascata is not atomic.
**How to avoid:** Use a Supabase RPC function or a database-level DEFAULT/TRIGGER for numero_transformacao. Alternatively, since cascata is called sequentially per user action and card_key collisions are rare, this is LOW risk. Document it as a known edge case.
**Warning signs:** Two cards with same card_key + same numero_transformacao.

### Pitfall 5: database.types.ts does not include transformacoes
**What goes wrong:** TypeScript errors when querying transformacoes table because the generated types don't include it.
**Why it happens:** The types file was generated before migration 00006 was applied, and was never regenerated. Current cascata route uses `(supabase as any)` workaround.
**How to avoid:** Regenerate database types with `npx supabase gen types typescript` OR continue using `as any` pattern consistently (existing convention). Given that cascata route already uses `as any`, maintaining this approach is acceptable for consistency.
**Warning signs:** TypeScript compilation errors on `.from('transformacoes')`.

### Pitfall 6: Atribuicoes table shared between prateleira and transformacao
**What goes wrong:** Assigning a separador to a transformacao card conflicts with the prateleira assignment for the same card_key.
**Why it happens:** The atribuicoes table uses card_key as the grouping key. A transformacao "card" (card_key + numero_transformacao) is not the same as the prateleira card (card_key only).
**How to avoid:** Use a composite key for transformacao atribuicoes: either add a `tipo` discriminator (e.g., 'separador_transformacao') or store a `transformacao_card_key` (e.g., "cardkey::transf::1"). The simplest approach is to use `card_key::transf::N` as the atribuicoes.card_key for transformacao assignments.
**Warning signs:** Assigning separador to transformacao overrides prateleira assignment.

## Open Questions

1. **Atribuicoes key strategy for transformacao cards**
   - What we know: atribuicoes table uses card_key. Transformacao cards need their own atribuicoes separate from prateleira.
   - What's unclear: Whether to use composite string key (`card_key::transf::N`) or add a new column to atribuicoes.
   - Recommendation: Use composite string key -- simpler, no migration needed for atribuicoes table, and the `::transf::` separator is unlikely to appear in real card_keys (which use `grupo_envio::tipo::importacao_numero` format). This is LOW risk.

2. **grupo_envio for transformacao cards**
   - What we know: Transformacao records have card_key but no grupo_envio. Need it for kanban columns and urgency.
   - What's unclear: Whether to store grupo_envio redundantly on transformacoes table or derive it from pedidos at query time.
   - Recommendation: Derive from pedidos via card_key join in the data hook. This avoids schema changes and stays consistent with existing data model.
